# THQLLM 官网与文档门户设计规格

- 日期：2026-07-13
- 状态：设计已确认，等待规格复核
- 域名：`thqllm.com`
- 形态：纯前端静态官网与 Markdown/MDX 文档门户

## 1. 项目目标

THQLLM 官网承担两个同等重要的职责：

1. 作为统一项目门户，让访客快速理解并进入 THQLLM 旗下的 AI 项目。
2. 作为统一文档入口，让每个项目都能通过仓库内 Markdown/MDX 维护独立使用文档。

官网需要有鲜明的东方 Project 同人弹幕游戏气质，但不使用官方角色、现成角色立绘或来源不清晰的同人素材。视觉表达依靠原创幻想场景、弹幕阵列、关卡编号、游戏选单、符卡式标签和同人游戏排版完成。

首版使用简体中文，代码结构、内容目录和导航模型保留后续国际化能力。

## 2. 非目标

首版不实现以下能力：

- 用户注册、登录和统一账号系统
- 论坛、评论、私信和社区审核
- 数据库或常驻 Node 服务
- 对外部项目进行 iframe 嵌入
- 跨站共享登录态或密钥
- 运行时在线状态监控
- 官方东方 Project 角色、标志、音乐或未经授权的同人素材

如果未来出现明确的社区需求，论坛应作为独立服务部署到单独子域名，不与静态官网耦合。

## 3. 开源底座决策

项目采用 **Rspress 2.0 + 自定义 React 主题**。

选择原因：

- 原生支持 Markdown/MDX、静态构建、全文搜索、代码高亮和国际化。
- 自定义 React 首页可以完整实现游戏标题画面，不受传统文档首页布局限制。
- 文档主题可以复用成熟导航能力，同时覆盖品牌色、项目切换器和局部组件。
- 构建结果为纯静态文件，可部署到 Nginx、Cloudflare Pages 或其他静态托管服务。
- Rspress 采用 MIT 许可证，适合作为长期维护底座。

不直接复用现有东方同人网站仓库。检索到的相关项目大多不是“项目门户 + 文档系统”模板，且常见维护不足、许可证缺失或视觉素材来源不清晰的问题。

## 4. 信息架构

| 路由 | 作用 |
| --- | --- |
| `/` | THQLLM 游戏标题式首页与核心项目入口 |
| `/projects/` | 完整项目目录、能力标签和外部访问入口 |
| `/docs/fluctgraph/` | FluctGraph 文档空间 |
| `/docs/thq-api/` | THQ API 文档空间 |
| `/docs/toho-image-studio/` | Toho Image Studio 文档空间 |
| `/notes/` | 开发札记与更新记录 |
| `/about/` | 关于 THQLLM、项目原则和联系信息 |
| `404` | 游戏 Continue 风格的静态错误页 |

外部项目保持独立域名：

- FluctGraph：`https://graph.tohoqing.com/`
- THQ API：`https://sub.thqllm.com/`
- Toho Image Studio：`https://img.tohoqing.com/`

官网不代理项目业务流量，只负责品牌呈现、项目发现和文档路由。

## 5. 内容模型

### 5.1 项目清单

所有项目由一个类型化配置文件维护：

```ts
interface ProjectDefinition {
  id: string;
  name: string;
  stageLabel: string;
  categoryLabel: string;
  description: string;
  externalUrl: string;
  docsBase?: string;
  accent: "vermilion" | "cyan" | "gold" | "sakura";
  tags: string[];
  order: number;
  featured: boolean;
}
```

配置在构建阶段进行结构校验。缺少项目名、合法 URL、排序值或必要描述时，构建失败，避免发布残缺入口。

`docsBase` 对首版三个项目为必填，对未来新增项目可选。未来尚未建立文档的项目仍可展示外部访问入口，但文档按钮显示“文档准备中”，不得生成空链接。

### 5.2 文档内容

文档按项目隔离：

```text
docs/
  fluctgraph/
  thq-api/
  toho-image-studio/
```

每个项目首版至少包含：

- 概览
- 快速开始
- 常见问题
- 更新记录

根据项目能力增加专属章节：

- FluctGraph：安装、AI IDE、Agent 工作流、MCP、LightRAG、配置参考
- THQ API：API Key、Base URL、模型列表、兼容接口、SDK/cURL、错误处理
- Toho Image Studio：访问配置、模型与引擎、图像生成、图像编辑、提示词片段、直连接口

文档只写入已确认的信息。无法从项目本身或用户材料确认的命令、参数和接口不得虚构。

### 5.3 新增项目流程

1. 在 `src/data/projects.ts` 登记项目元数据。
2. 如需文档，复制项目文档目录模板并编写 Markdown/MDX。
3. 构建时自动更新首页关卡、项目目录、文档导航和搜索索引。

## 6. 品牌语言

全站唯一主标题为 **THQLLM**，不使用额外中文副品牌名。

首页固定文案体系：

- 英文说明：`AI PROJECTS · TOOLS · EXPERIMENTS`
- 中文说明：`模型中转 · AI 编程 · 图像生成`
- 项目入口：`PROJECT SELECT / 项目选择`
- 文档入口：`MANUAL / 使用文档`
- 开发札记：`EXTRA STAGE / 开发札记`
- 关于页面：`OMAKE / 关于 THQLLM`

不使用“智能结界”“结界”或意义相近的生造世界观名词。东方气质由视觉和游戏交互语言承担，功能文案保持直接。

## 7. 首页体验

### 7.1 首屏

首页首屏直接呈现可操作的游戏标题画面，不设置营销式 Hero 卡片或长篇品牌介绍。

首屏组成：

- THQLLM 主标题
- 中英文项目类型说明
- 四项游戏选单式导航
- 原创幻想场景背景
- 低密度弹幕环与轻量 HUD
- 当前项目数量和版本信息
- 下一段项目区域的可见提示

首屏高度使用受控的响应式范围，所有桌面和移动视口都应露出下一段内容。

### 7.2 场景资产

最终首屏使用原创、无人物的横向位图场景，包含远山、雾、鸟居或神社建筑轮廓等东方幻想元素。该场景不得复制东方 Project 官方画面，也不得使用来源不明的角色或同人插画。

场景图只承担氛围。标题、导航和项目状态必须使用真实 HTML 文本，图片加载失败时仍可完整操作。

### 7.3 动效

- 弹幕以低速、低密度方式环绕，不穿过主要按钮。
- 雾层和远山允许轻微视差，只使用 `transform` 和 `opacity`。
- 选单反馈参考同人弹幕游戏，但不自动播放音乐或音效。
- `prefers-reduced-motion` 开启时停止循环动画和视差。
- 移动端隐藏 HUD、降低弹幕数量并改用单列选单。

### 7.4 首页内容顺序

1. 游戏标题式首屏
2. 三个项目的 Stage Select
3. 各项目文档快捷入口
4. 最新开发札记
5. 关于 THQLLM 的简短说明与页脚

项目卡片是重复内容单元，不嵌套其他卡片。每个项目始终提供两个分离操作：进入项目、阅读文档。

## 8. 视觉系统

### 8.1 色彩

使用多色但受控的东方同人游戏配色：

- 墨绿：`#263E3A`
- 朱红：`#A92935`
- 纸白：`#FFFDF7`
- 樱粉：`#F28AA0`
- 青色：`#70D8C7`
- 金色：`#F4CF71`
- 天青：`#72AEBC`

页面不得被单一紫色、深蓝、米色或红色支配。文档正文以纸白和墨色为主，彩色只用于层级、项目识别和状态提示。

### 8.2 字体

- THQLLM 标题：高对比英文衬线字体
- 中文标题：开源中文衬线字体或可靠系统宋体回退
- 正文和界面：清晰的无衬线字体
- HUD、编号和代码：等宽字体

字体必须自托管或具有可靠的系统回退，不依赖在中国大陆访问不稳定的远程字体服务。

### 8.3 形状与布局

- 卡片圆角不超过 `6px`
- 不使用渐变球、光斑球或装饰性浮动卡片
- 不在卡片中嵌套卡片
- 页面区块使用完整横向带状布局
- 图标按钮优先使用 Lucide 图标并提供工具提示
- 所有固定格式控件使用稳定尺寸，动态内容不得造成布局跳动

## 9. 文档体验

文档页延续 THQLLM 品牌，但正文优先服务长时间阅读。

桌面布局：

- 顶部：THQLLM、当前项目、项目切换器、全文搜索和外部项目按钮
- 左侧：当前项目的独立文档导航
- 中间：宽度受控的正文
- 右侧：当前页面目录

移动布局：

- 左侧导航改为抽屉
- 页内目录收起
- 搜索入口保留
- 正文不出现横向滚动，代码块自身可滚动

首版 MDX 组件限制为：

- 提示框
- 步骤列表
- API 端点
- 参数表
- 代码分组
- 项目外链按钮

不提供任意布局组件，避免文档演变成难以维护的页面搭建系统。

全文搜索覆盖所有项目，并在结果中明确标注所属项目。站点同时生成适合搜索引擎和 AI 工具读取的基础元数据、站点地图及 `llms.txt`。

## 10. 异常处理与降级

- 项目配置错误在构建阶段失败。
- 暂无文档时显示明确状态，不创建空路由。
- 外部项目不显示未经可靠验证的在线状态。
- 外部链接使用安全的新窗口属性。
- 场景图加载失败时显示稳定背景色，文字和导航保持可见。
- JavaScript 不可用时，核心导航、项目列表和文档正文仍可访问。
- 自定义 `404` 页面提供返回首页、项目目录和搜索操作。

## 11. 无障碍与响应式要求

- 所有选单和按钮均支持键盘操作。
- 焦点状态清晰，不只依赖颜色表达。
- 正文、按钮和代码块满足可读对比度。
- 装饰场景和弹幕不进入无障碍树。
- 页面缩放至 200% 时不遮挡主要功能。
- 验证至少 `1440x900`、`1024x768`、`390x844` 和 `360x800` 四类视口。
- 中文长标题、英文长单词和项目标签不得溢出容器。

## 12. 测试策略

### 12.1 单元测试

使用 Vitest 覆盖：

- 项目配置校验
- 项目排序与筛选
- 文档路由生成
- 导航数据生成
- 关键交互组件状态

### 12.2 浏览器测试

使用 Playwright 覆盖：

- 首页选单和键盘导航
- 三个项目外链
- 项目文档入口
- 项目切换器
- 全文搜索
- `404` 返回路径
- 桌面和移动端导航
- 减少动态偏好

### 12.3 构建与视觉验证

- 执行完整静态构建
- 检查 Markdown 内链和资源路径
- 检查 HTML 标题层级和基础无障碍
- 对首页和文档页进行桌面、移动端截图回归
- 检查首屏非空、场景构图正确、文字无重叠、动画不遮挡操作

## 13. 部署

构建输出目录为静态文件目录 `doc_build/`。

部署要求：

- 不依赖数据库
- 不依赖运行时 Node 服务
- 支持 Nginx 或静态托管平台
- 配置 `thqllm.com` 的 HTTPS、缓存和压缩
- HTML 使用短缓存，带哈希的静态资源使用长期缓存
- 部署前必须通过测试、链接检查和生产构建

## 14. 首版验收标准

首版完成必须同时满足：

1. 首页主标题只有 `THQLLM`，全站不出现“智能结界”或“结界”。
2. 首页具有明确的东方同人弹幕游戏气质，但不含人物和未经授权的 IP 素材。
3. 三个项目均有正确的外部入口和独立文档空间。
4. 新增项目只需要更新一个项目配置并按需添加 Markdown。
5. 文档支持独立侧边栏、页内目录、代码高亮和跨项目搜索。
6. 桌面与移动端不存在文字遮挡、按钮溢出和失效入口。
7. 减少动态、图片失败和 JavaScript 不可用时仍能完成核心导航。
8. 项目可成功生成纯静态构建产物。
9. 自动化测试、链接检查和视觉验证全部通过。
